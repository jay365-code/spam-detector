# 스팸 탐지 시스템 - IBSE (시그니처 추출) 데이터 파이프라인 정밀 분석

이 문서는 스팸 메시지가 인입된 후, 최종적으로 고유 시그니처가 적재되기까지의 전 과정을 담고 있습니다. 
IBSE 시스템의 핵심 설계 사상은 **"LLM의 연산 비용은 극도로 아끼고 성능은 최대로 뽑는다"**는 데에 있으며, 이는 다음의 가장 강력한 **3 Step 여과망 순서(DB -> Cache -> LLM)**에 기반합니다.

---

## 1. 최적화된 3 Step 시그니처 판별 및 추출 파이프라인

스팸 판정을 받은 메시지가 시그니처 추출망에 들어올 때, 시스템은 무작정 LLM 에이전트부터 돌리지 않고 철수한 연산 절약 테크트리를 탑니다.

### 🥇 Step 1: 영구 스토리지 캐싱 (`signatures.db` Hit)
- **위치**: 백엔드 시스템 최전방 (`SignatureDBManager` 등)
- **로직**: 이 메시지가 과거에 이미 누군가에 의해 적발되어 `signatures.db`에 등록된 시그니처를 포함하고 있는지(초고속 SQLite `instr` 쿼리) 검사합니다.
- **결과**:
  - **Hit**: 즉시 해당 시그니처를 재활용하여 차감(판단) 완료. (초고속 탈출)
  - **Miss**: "DB에 없는 패턴"으로 판단하여 Step 2로 이동.

### 🥈 Step 2: 실시간 인메모리 캐싱 (Session `Cache Hit`)
- **위치**: `IBSEAgentService` 진입 직후 (`signature_cache`)
- **로직**: DB에는 없지만, **"방금 전 같은 엑셀 파일이나 같은 세션을 돌리면서 LLM이 막 뽑아둔 따끈따끈한 시그니처"**가 포함된 중복 변종 텍스트인지 검사합니다.
- **결과**:
  - **Hit**: 방금 전 뽑은 시그니처를 동일하게 재활용 부여(`decision: use_string (cached)`). 역시 LLM 연산을 우회합니다. 다량의 비슷한 엑셀 스팸 처리 시 API 요금을 폭발적으로 구원하는 핵심 단계입니다.
  - **Miss**: "기존 DB 에도 없고 방금 작업한 내역에도 없는 **역사상 최초의 변종**"으로 최종 규정되어 마지막 Step 3로 인계됩니다.

### 🥉 Step 3: LLM 다이렉트 추출 및 교정 (`agent.py` Graph)
- 위 두 겹의 방어막과 필터를 당당하게 통과한 '진짜 엑기스 신종 스팸'들만이 드디어 `ibse_graph` 워크플로우를 타게 됩니다. 비싼 API 비용을 지불할 가치가 있는 녀석들입니다.

---

## 2. 미시적 구조: Step 3 (LLM 추출) 의 강박적 검증 엔진

Step 3에 들어온 데이터들은 예전처럼 후보군을 쪼개지 않고, 한 번의 강력한 LLM 추론과 깐깐한 파이썬 로직에 의해 깎여 나갑니다.

### 2.1 다이렉트 추출 노드 (`selector.py`)
- LLM 본연의 통찰력으로 문장 전체(`original_text`)와 난독화가 해제된 부분(`obfuscated_urls`) 등을 읽게 하여, 가장 스팸 식별력이 높은 치명적 부위를 단독 발췌하게 합니다.
- **Python 기반의 안전 확장 조치 (트런케이션)**:
  - **URL 앵커 보존**: 발췌된 구간에 URL이 포함되어 있다면, 절대 도중에 잘리지 않게 중심점(Anchor)으로 보존하며 남은 여유 공간만 늘립니다.
  - **Dead Zone 강제 패딩**: `use_sentence` 판별 시 39바이트에 미달하면, 주변 원문 글자를 하나씩 붙여가며 억지로 규격(39~40 Bytes)을 완성시킵니디. (불가 시 `use_string`으로 강등)
  - **CP949 모지바케 회피**: 배열 문자 단위가 아닌 바이트 연산(`encode('cp949')`) 단위로만 슬라이싱하여 한글 가변 바이트가 잘려 깨지는 장애를 근본적으로 차단합니다.

### 2.2 극한의 철칙 노드 (`validator.py`)
 추출물은 무조건 이 노드를 통과해야 합니다.
1. **URL 단독 꼼수 추출 봉쇄**: 추출된 조각이 딱 URL 부분에만 종속되어 있으면 차단합니다. 주변부의 한글(조사나 텍스트)이 반드시 결합되어야만 고유성을 통과시킵니다.
2. **할루시네이션(창작) 검거**: 추출된 시그니처가 공백 없는 원본 텍스트에 한 글자도 틀림 없이 존재하는 서브스트링인지(`in` 연산자) 확정 검사합니다.
3. **블랙리스트 차단**: `무료거부`, `080-` 등 스팸이 아니어도 쓰는 평범한 문구가 단독 조각으로 나오면 즉결 파기.
4. **극단적 바이트 룰**: `use_string`은 9\~20, `use_sentence`는 오직 39\~40바이트 통과.

### 2.3 순환 루프 (Self-Repair)
- 만약 검증(`validator`)에 막히면 에러 사유를 안고 돌아가 다시 뽑아오라고 시키는 **자가 로컬 수리(Self-Repair)** 그래프 구조(노드 사이클)가 완성되어 있습니다.
- 모든 루프와 검증을 당당히 통과한 추출물은 결과 엑셀에 반영되는 동시에, **Step 2의 인메모리 캐시 방어망에 실시간으로 장전되며**, 관리자의 검토 이후 **Step 1의 DB 스토리지 장비로 완전 이관**되어 사이클을 수호합니다.
