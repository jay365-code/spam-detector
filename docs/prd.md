## 📝 Product Requirements Document (PRD)

### 1. 개요 (Introduction)

본 문서는 LangChain Agent와 Antigravity 모델을 활용한 **문자 메시지 스팸 Detector 시스템**의 요구사항을 정의합니다. 본 시스템은 웹 인터페이스를 통해 대량의 메시지 데이터를 처리하며, **3단계 하이브리드 필터링**을 통해 분류 정확도와 처리 효율성을 극대화합니다.

#### 1.1. 시스템 목표
* **정확성:** 규칙(Rule), **RAG 기반 AI(Antigravity + Guide)**, 심층 도구(URL)를 결합하여 오탐/미탐 최소화.
* **편의성:** 엑셀 파일 업로드/다운로드 기반의 웹 UI 제공.
* **상세 분류:** 단순 스팸 여부뿐만 아니라, 가이드라인에 근거한 **표준 분류 코드**를 식별하여 기록.

### 3. 기능 요구사항 (Functional Requirements)

#### 3.1. 웹 인터페이스 및 데이터 처리
* **FR-3.1.1 (파일 처리):** `.xlsx` 파일의 "시뮬결과전체" 탭, "메시지" 컬럼 데이터를 읽어옵니다.
* **FR-3.1.2 (결과 출력 포맷):** 원본 엑셀 파일에 다음 두 컬럼을 추가하여 결과를 기록합니다.
    * **"구분"**: 최종 판정 결과 (**SPAM**, **HAM**, **UNKNOWN** 중 하나).
    * **"분류코드"**: 하단 **[3.3 분류 코드 표준]**에 정의된 번호 기입 (SPAM인 경우 필수).

#### 3.2. 핵심 분류 아키텍처 (Flow Control Logic)

Agent는 다음의 **3단계 파이프라인**을 준수해야 합니다.

**단계 1: 1차 필터 - 규칙 기반 감지 (Rule-Based Detection)**
* **목표:** 명확한 패턴 매칭을 통한 신속 처리.
* **로직:**
    * **HAM 감지:** `[HAM으로 분류]` 기준(1:1 대화 등) 매칭 시 -> **HAM 확정** (단계 종료).
    * **SPAM 감지 (특수문자 난독화):**
        * [cite_start]**조건:** 글자 사이에 의미 없는 특수문자나 기호가 삽입된 패턴 감지 ("향.꼼.썽", "안/내/주" 등)[cite: 14, 15].
        * **행동:** 패턴 발견 시 **SPAM 후보로 식별하되, 상세 분류 코드 식별을 위해 단계 2로 이관.**
    * **판단 불가:** 위 패턴 미발견 시 -> **단계 2로 이관.**

**단계 2: 2차 필터 - RAG 기반 유형 분석 (RAG & AI Analysis)**
* **목표:** 가이드 문서를 참조하여 스팸의 분류 코드를 식별하고 애매한 문맥을 분석.
* **기준:** **"스팸 및 햄 분류 가이드 문서"** 내의 SPAM 분류 항목.
* **로직:**
    * **Context Retrieval (RAG):** 메시지와 유사한 가이드 조항을 Vector DB에서 검색.
    * **Case A (1차 필터에서 난독화 감지됨):**
        * RAG 검색을 통해 메시지가 해당하는 **분류 코드(1~10)**를 매핑.
        * **결과:** **SPAM** + **분류코드** 확정. (**단계 3 건너뜀**)
    * **Case B (1차 필터에서 판단 불가):**
        * Antigravity 모델 확률 + RAG 검색 결과를 종합 판단.
        * **SPAM 매칭 시:** 가이드 규칙에 부합하면 **SPAM** + **분류코드** 확정. (**단계 3 건너뜀**)
        * **HAM 매칭 시:** 가이드 규칙에 부합하면 **HAM** 확정.
        * **여전히 애매함:** **단계 3으로 이관.**

**단계 3: 3차 필터 - 심층 분석 (URL Deep Dive)**
* **조건:** 1, 2차 필터에서 확정되지 않은 애매한 메시지 중 URL이 포함된 경우.
* **로직:**
    * 메시지 내 URL을 추출하여 실제 접속(Scraping)을 시도.
    * [cite_start]**접속 성공 시:** 첫 페이지에 **회사소개 정보(사업자번호, 연락처, 주소 등)**가 없으면 **SPAM** 확정[cite: 18].
    * [cite_start]**접속 불가 시:** **TLD**가 정상(.com/.co.kr/.gov/.or.kr/.org)이면 **HAM**, 그 외는 **SPAM** 확정[cite: 19].
    * **분석 실패:** 판단 불가 시 **UNKNOWN** 처리.

### 4. 기술 스택 (Technical Stack)

| 구분 | 기술 스택 | 상세 역할 |
| :--- | :--- | :--- |
| **Frontend** | HTML/JS | 파일 업로드 및 결과 다운로드 UI |
| **Backend** | Python (Flask/FastAPI) | API 서버, 엑셀 핸들링 |
| **Orchestration** | **LangChain** | 3단계 Flow 제어, RAG Chain 구성 |
| **RAG** | **Vector DB** | 가이드 문서 임베딩 저장 및 유사도 검색 |
| **Model** | **Classification Service (Antigravity)** | Antigravity 플랫폼/엔진을 통해 메시지의 스팸 확률($P_{spam}$)을 계산하여 LangChain Agent에 제공하는 서비스 계층. |
| **Analyzer** | LLM (OpenAI/Gemini) | 가이드라인 기반 상세 분류 코드(1~10) 추론 |
| **Tools** | Python Request/BS4 | URL 접속, HTML 파싱(사업자정보 확인), TLD 추출 |

### 5. 비기능 요구사항 (NFR)

* **효율성:** 1차 및 2차 필터에서 SPAM이 확정될 경우, 고비용 작업인 3차 필터(URL 접속)를 수행하지 않아야 합니다.
* **데이터 무결성:** 엑셀 파일의 행(Row) 순서가 유지되어야 하며, 결과 컬럼이 정확히 매핑되어야 합니다.
* **보안:** 업로드된 데이터는 분석 완료 후 즉시 삭제 처리합니다.
