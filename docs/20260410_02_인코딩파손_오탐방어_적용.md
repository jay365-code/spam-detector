# Implementation Plan: 텍스트 인코딩 파손 치환 버그(물음표 난독화 오탐) 수정

## Issue 명세 (4월 9일자 분석 결과 등)
- **현상:** 원본 MMSC 로그의 인코딩(EUC-KR 등) 문제나 지원하지 않는 이모지로 인해 텍스트가 `????????` 형태나 `` 형태로 깨져서 들어오는 경우가 다수 발생함.
- **모순점:** 실제로는 단순한 시스템 인코딩 에러로 파손된 정상(HAM) 메시지임에도 불구하고, Content Agent(LLM)는 가이드라인을 너무 충실히 따른 나머지 이를 **"심각한 텍스트 난독화(물음표 나열)로 목적을 숨긴 피싱 스팸(Code: 2)"**으로 오판해버림.
- **최종 결과:** 억울한 정상 메시지가 스팸으로 차단되는 오탐 지표 상승 및, 의미 없는 파손 문자열 분석에 불필요한 LLM API 비용(Quota)이 소모됨.

## 구조적 원인 분석 (Root Cause)
LLM은 주어진 텍스트 환경 내에서 판단할 뿐, 이것이 엑셀 추출 서버의 '인코딩 오류'인지, 아니면 실제 스패머가 입력한 '난독화 트릭'인지 구분하기 어렵습니다. 따라서 모델에게 판단을 위임하는 것보다 파이프라인의 최전선(Python 로직)에서 **데이터 정합성 필터(Guardrail)**를 적용하는 것이 가장 직관적이고 비용 효율적인 해결책입니다.

### 1. `app/graphs/batch_flow.py` - 최전방 인코딩 깨짐 필터 추가
**수정 위치:** `content_node` 함수 시작 부분 (대략 57라인, LLM을 호출하는 `content_agent.acheck()` 직전)

```python
     async def content_node(state: BatchState):
         msg = state["message"]
         s1 = state.get("s1_result") or {}
         prefetched = state.get("prefetched_context")
         cb = state.get("status_callback")
         
         loop = asyncio.get_running_loop()
         if cb: await cb("🧩 [Unified Flow] Content Agent 의도 분석 노드 진입")
         
+        # --- [NEW] Pre-filter: 인코딩 파손(????, ) 방어선 ---
+        import re
+        clean_msg = re.sub(r'\s+', '', msg)
+        # 1. 물음표(?) 나 깨짐 기호()가 연속 10자 이상 등장하는가?
+        has_consecutive_broken = bool(re.search(r'[?]{10,}', clean_msg))
+        # 2. 혹은 전체 문자 중 파손된 문자의 비율이 40%를 초과하는가? (짧은 문자열 대비)
+        ratio_broken = (clean_msg.count('?') + clean_msg.count('')) / max(len(clean_msg), 1)
+        
+        if has_consecutive_broken or ratio_broken > 0.4:
+            if cb: await cb("🛡️ [System Filter] 인코딩 파손(예: ???) 텍스트 감지. LLM 난독화 오탐 방지를 위해 조기 통과 (HAM)")
+            res = {
+                "is_spam": False,
+                "classification_code": None,
+                "reason": "🛡️ [인코딩 파손 예외] 다량의 깨진 문자(?, )가 감지되어 LLM 난독화/오탐 방지를 위해 정상(HAM) 처리됨.",
+                "spam_probability": 0.0
+            }
+            return {"content_result": res}
+        # ----------------------------------------------------

         if prefetched:
             res = await content_agent.acheck(msg, s1, status_callback=cb, content_context=prefetched)
```

### 2. 기대 효과 (Expected Behavior)
- `????????????????????????` 와 같이 인코딩이 파손된 케이스가 유입되면 파이프라인에서 AI(LLM)에 쿼리를 날리기 전에 Python 단에서 선제적으로 컷오프(Cut-off) 합니다.
- 스팸 판정을 하지 않고 무조건 **✅ HAM (정상)**으로 반환하므로 키워드 기반 난독화 오탐(Code: 2) 이슈가 원천 차단됩니다.
- 한 건당 모델의 분석에 소요되는 시간 3~5초 및 API 요금(Quota) 부하를 방지할 수 있습니다.
- (참고) 만약 파손된 메시지 안에 교묘하게 악성 URL이 숨겨져 있었다 하더라도, 이어지는 URL Agent 가 정규식으로 이를 낚아채어 접속 검사를 하므로 보안상 누수도 없습니다.
