# Implementation Plan: 은닉 도메인(`dbb73.`) 추출 누락 오류 및 프롬프트 강화

## Issue 명세
- **원본 메시지:** `. .록후 보. .택 . .세요 . : 623242 dbb73.`
- **현상:** 기존 시스템(Content Agent Prompt)에는 확연히 `nike26.` 이나 `dbb73.` 처럼 확장자가 생략된 문자열을 `.com`으로 복원하여 `obfuscated_urls`에 넣도록 하는 지시어가 이미 존재합니다.
- **모순점:** 그럼에도 불구하고 LLM은 이 단어를 도메인으로 인식하지 못하고 "순수 파편화된 극단적 난독화 문자열"이라 여겨 **HOLD_SHORT (관찰 대기)** 판정을 내렸습니다. 이에 따라 URL 리스트가 비어있게 되었고, URL Agent는 호출조차 되지 않고 통과(`URL 없음`)되었습니다.
- **원인:** 전체 문장 내에 마침표(`.`)가 너무 많이 혼재되어 있어(`보. .택 . .`), LLM이 문맥을 분석할 때 마지막의 `dbb73.` 역시 도메인 단서가 아닌 **단순 난독화를 위한 의미 없는 구두점의 연장선**으로 착각(Hallucination)한 것입니다.

## 구조적 원인 및 대책 (Root Cause & Solution)
이 문제는 Python 정규식을 조작하기보다는, `Content Agent`의 시스템 프롬프트(LLM 판단 기준)에서 **최우선 판단 원칙(우선순위)**을 강제함으로써 해결해야 합니다. LLM이 `HOLD_SHORT`를 결정하기 전에 은닉 도메인 유무를 무조건 먼저 의심하도록 강제 가이드라인을 삽입해야 합니다.

## 🛠️ 제안하는 코드 개선책 (Proposed Changes)

개발 서버 환경에서 `app/agents/content_agent/agent.py` 파일 내 시스템 프롬프트를 다음과 같이 직관적이고 강력하게 수정하십시오.

### 1. `app/agents/content_agent/agent.py` - 프롬프트 룰 강화
`HOLD_SHORT` 판단 기준 및 은닉 도메인 복원 룰셋 부분에 강력한 예외 지시어를 붉은 글씨(강조) 뉘앙스로 추가합니다.

**수정 위치:** `Step 3. [은닉 도메인 (Covert Domain) 복원 및 추출]` 및 `HOLD_SHORT` 조건부 (대략 670라인 부근)

```diff
     Step 3. [은닉 도메인 (Covert Domain) 복원 및 추출]
     ...
     - 👉 확장자 생략/점찍힘: `NH1245`, `메인337`처럼 뜬금없는 코드나, `nike26. `처럼 확장자가 빠진 경우 `.com`을 붙여 복원하라 (예: `NH1245.com`, `nike26.com`).
+    - 🚨 [★절대 주의★]: 메시지 내에 마침표(.) 등 특수문자가 난잡하게 섞여 있더라도, 영문자와 숫자로 조합된 단어 뒤에 점이 붙은 형태(예: `dbb73.`)가 보인다면 이는 무의미한 점이 아니라 100% 은닉 도메인(`dbb73.com`)입니다!
+    - 위와 같은 은닉 도메인 단서가 단 하나라도 발견되면, 절대로 `HOLD_SHORT` 판정을 내리지 말고 무조건 `SPAM (Code: 2)`으로 차단한 뒤 `obfuscated_urls`에 복원된 도메인(`dbb73.com`)을 반드시 기재하십시오.

     ...
     "label": "HAM|SPAM|HOLD_SHORT",
     - URL 등 명백한 접근 유도 없이 짧은 단어 파편만 존재할 때 HOLD_SHORT 적용.
+    - 단, 파편화 문자 사이에서도 `영문+숫자+마침표` 형태의 은닉 도메인(`dbb73.`)이 식별되면 HOLD_SHORT를 취소하고 SPAM으로 통과시킬 것!
```

### 2. 기대 효과 (Expected Behavior)
- 문장 안에 마침표가 수십 개 찍혀있는 극단적 난독화 상태라도, LLM은 `영문+숫자` 조합 뒤에 마침표가 오는 특정 패턴(`dbb73.`)을 발견하는 즉시 시선을 집중하게 됩니다.
- 단순 파편화(HOLD_SHORT)로 치부하던 오판을 그만두고, "아 이건 `dbb73.com`으로 유도하는 은닉 URL이구나"라고 깨달아 `SPAM` 판정과 함께 URL을 추출해 넘겨줍니다.
- 넘겨진 `dbb73.com`은 정상적으로 Router 노드를 통과(has_url=True)하여 URL Agent의 강력한 스크래핑 검사를 받게 됩니다.
