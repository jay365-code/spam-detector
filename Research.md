# URL 처리 Flow 정밀 분석 로그 (Research.md)

이 문서는 AI 기반 스팸 메시지 탐지 시스템에서 "입력 텍스트의 URL 처리"가 어떻게 흘러가는지(`batch_flow.py`, `url_agent/nodes.py` 등 기준) 정밀 분석한 내용입니다.

## 1. 개요
스팸 탐지기 파이프라인에서 URL 분석은 가장 강력하면서도 복잡한 검증 단계입니다. 메시지 파편화 및 난독화, 위장(Decoy) 방패막이 도메인, IP 우회, 커스텀 슬러그 등 다양한 스팸 차단 회피 기술을 방어하기 위해 다계층(Multi-Layer) 필터 및 검증 아키텍처로 구현되어 있습니다.

전체적인 Flow는 크게 5가지 스텝으로 구성됩니다.
1) **사전 수집 및 오염 확인 (Pre-filtering)**
2) **정밀 문자열 기반 추출 (Regex & De-obfuscation)**
3) **다중 에이전트 캐시 및 조기 판정 (Vanguard & Caching Layer)**
4) **스크래핑 및 시각/HTML 분석 (Agent Execution)**
5) **최종 병합 및 환각/정책 필터링 (Aggregation & Override)**

---

## 2. 세부 처리 플로우

### Step 1: 사전 수집 및 오염 확인
`batch_flow.py` 의 `url_node` 진입 시 가장 먼저 처리되는 단계입니다.
- **KISA 파리미터 활용 (`pre_parsed_url`)**
  - KISA 등에서 추출하여 넘겨준 분리된 URL 파라미터가 있다면 우선 검토합니다.
  - **단축 URL 파손 방어선:** 전달받은 도메인이 `bit.ly`, `vo.la`와 같은 단축 URL 형식인데도 파라미터 부분이 비어 있거나 손상(괄호, 별표 등)된 경우, KISA 추출 오류로 간주하여 강제로 파라미터를 폐기하고 "본문 텍스트 강제 추출 모드"로 전환합니다.

### Step 2: 정밀 추출 및 보정 (extract_node)
`agents/url_agent/nodes.py` 의 `extract_node`에서 본문을 기반으로 URL을 직접 발라냅니다.
- **난독화 복원 융합:** Content Agent가 정규화/디코딩 과정에서 찾아낸 `obfuscated_urls` 리스트를 최상우선순위로 삽입합니다.
- **정규식 다중 스캔:** 
  1. `http/https` 프로토콜 기반 명시적 URL 
  2. 도메인 형식으로 된 텍스트 (프로토콜 생략형)
  3. 띄어쓰기가 의도적으로 누락/조작된 버전의 텍스트 스캔(`spaceless_message` 등)
- **오탐 방지 및 정제 (Gluing 방어 & Garbage Tuning)**
  - **TLD 병합 해제:** `.한국`, 영문 TLD 뒤에 한글 문장이 실수로/고의로 붙은 경우 분리(`youtube-dm.com재무상담` -> `youtube-dm.com` 분리).
  - **커스텀 슬러그(경로) 보호:** 특정 단축 URL 뒤쪽의 한글 경로가 기호 뒤에 나오는 경우 보존하되, 스팸성 식별자(예: `code:`, `tel:`, 구두점 등)만 정밀 타격하여 제거합니다.
  - **Intelligent Sorting:** 무거운 일반 도메인 검사를 먼저 수행하도록, 단축 URL이나 UGC(사용자 생성 컨텐츠, 오픈카톡 등) 도메인은 검사 배열의 뒤쪽으로 미루어 배치 효율을 극대화합니다.

### Step 3: 조기 판정 및 스마트 캐시 (Vanguard 3-Strike)
자원 고갈을 막고 봇 차단 로직에 대응하기 위한 배치 그룹(Batch Flow) 내 런타임 최적화 구간입니다.
- **영구 화이트리스트 스캔 (`UrlWhitelistManager`):** 영구로 저장된 안전 DB에 도메인이 있으면, 추가 검사 없이 즉초고속 통과(HAM) 처리됩니다. (단, 오픈카카오, 텔레그램 등의 UGC 도메인은 제외)
- **Vanguard 3-Strike 룰 (런타임 릴레이 셰어):** 짧은 시간 내에 여러 스레드가 동시에 같은 피싱 도메인을 스크래핑하면 서버 차단이 발생합니다.
  - 1번 타자(리더/Vanguard)가 선발되어 먼저 해당 URL 분석에 진입하고 락(`Condition`)을 켭니다. 
  - 나머지 에이전트들은 대기하며 리더의 결과를 기다립니다(Runtime Cache).
  - 리더가 스팸/에러로 실패하면 `Strike`가 증가하며, 3 Strike 도달 시, 이후 대기자는 더 이상 시도하지 않고 즉각 `SPAM/차단` 확정으로 무한 대기를 종료합니다.

### Step 4: 스크래핑 및 시각-HTML 분석
`url_agent` 내부의 동적 스크래핑과 LLM 분석 노드입니다.
- **시각 정보 활용 (Vision API):** HTML이 암호화되거나 DOM이 의도적으로 빈약한 경우 스크린샷 기능을 활용합니다 (PlaywrightManager).
- **Gemini / Claude Fallback:** Qouta Limit (429)이나 타임아웃 발생 시, 설정된 예비 키(Pool)나 다른 대체 서브 모델을 이용하는 자동 재시도 로직이 포함되어 있습니다.
- **신뢰 불일치 (Mismatched):** 사이트는 안전해 보이지만 문자 내용과 괴리가 큰 "가짜 위장 목적 (방패막이)" 인지 면밀히 스캔합니다.

### Step 5: 최종 병합 및 필터링 (aggregator_node)
가장 중요한 논리 계층(`batch_flow.py` 의 `aggregator_node`)입니다. Content Agent의 결과와 결합하여 최종 판결(Verdict) 및 오류 수정을 진행합니다.
- **AI 환각 무결성 퓨전 (Hallucination Defense):** LLM이 URL을 가상으로 생성하지 않았는지 실제로 원문 또는 디코딩 텍스트에 들어있는 도메인인지 크로스 체크(`is_url_in_message`)합니다. 가짜 IP형태의 응답도 이곳에서 차단됩니다.
- **분리 감지(Red Group) 트리거:** 텍스트가 정상(HAM)으로 보이지만 숨겨진 URL이 스팸인 경우, "단순 URL 스팸" 격리로 간주하여 엑셀 리포트에서 적색 바탕으로 분리 표기(Red Group)합니다. (상거래와 완벽히 맥락이 일치하면 HAM 면책권 발행)
- **방패막이(Decoy) 배제 조치 (`drop_url`):** 정상 도메인 위장, 데드링크, 쓸모없는 파편형 단축 URL로 판명된 경우 최종 엑셀의 URL 열에 표출되지 않도록 과감히 삭제 조치를 취합니다.
- **중복 시그니처 배제:** 추출된 URL 파편이 살아남을 경우 이미 URL단에서 걸러졌으므로, 엑셀 칸 낭비와 중복을 막기 위해 IBSE(시그니처)에서 해당 텍스트를 제거(`unextractable` 처리)합니다.

### Step 6: IP 스타일 URL 배제 조치 (버전/숫자 나열 오탐 강력 통제)
시스템 내에서 숫자와 점(.)으로 구성된 일련의 문자열은 일반 어플리케이션 버전(예: `앱버전 1.4.7.9`)이나 단순 숫자의 나열임에도 URL로 잘못 추출되는 심각한 오탐 이슈가 존재합니다. 따라서 이를 원천 차단하기 위한 하드코딩된 정규식 필터가 `aggregator_node`에 구현되어 있습니다.
- **정규식 필터링:** 도메인 파츠가 `^\d{1,3}(\.\d{1,3}){3}$` 형태(IP 주소 형태)와 완벽히 일치(`1.4.7.9` 등)하는지 정밀 스캔합니다.
- **강력한 정책 적용:** 공인 IP, 사설 IP 여부와 무관하게 시스템 상 모든 IP 주소 형태는 URL 추출 대상에서 **"강제 배제(Drop)"** 되며, 악의적인 목적의 IP 우회 URL조차 엑셀 최종본에서는 표출되지 않도록 과감하게 삭제(가짜 IP 판단, `fake_ip`)하는 정책을 취합니다.

## 3. 요약 시사점 (Key Takeaways)
1. **견고한 텍스트 처리:** 정규식에 의존하지 않고, 띄어쓰기가 생략되거나 `.한국` 같은 Punycode, 스팸성 꼬리표(`code:` 등)를 전처리하는 **Heuristic String Manipulation** 기법이 매우 발달되어 있습니다.
2. **비용 및 리소스 극강화 (Vanguard Pattern):** 여러 피싱 URL 쿼리가 쏟아질 때 동시 다발적 크롤링 스파이크를 막고, 단 3회의 실패로 전체 배치 군을 안전하게 우회시키는 아키텍처가 돋보입니다.
3. **무죄추정 폐기 / Red Group 방어:** 텍스트 검사를 이탈한 위장 URL 스팸들에 대해 강제 스팸 마킹(Red Group)을 부여하고, "도메인이 깨끗하더라도 맥락이 맞지 않으면 방패막이 스팸이다" 라는 엄격한 비즈니스 룰이 코드 레벨에서 통제되고 있습니다.
4. **숫자형 오탐 철저 차단:** IP 주소 형식(`1.4.7.9` 등)의 경우 URL 파이프라인에서 무조건 배제시켜 부가적인 버전 정보/숫자 나열이 스팸 증거(URL)로 기록되는 일을 원천 봉쇄하고 있습니다.
